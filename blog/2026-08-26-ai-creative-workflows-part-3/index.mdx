## Meet Foundry

Throughout this series, we are building a small open-source application called Foundry.

Foundry is not trying to replace Dropbox, Google Drive, or your existing asset manager. It adds something those tools were never designed to do: understand what files actually contain.

It indexes existing creative folders and makes them searchable by meaning. No manual migration. No requirement to reorganize years of historical projects before the system becomes useful.

Foundry should work with the files teams already touch every day: concept art, PSD files, Blender scenes, video exports, audio, PDFs, references, and production docs.

## Phase 1 — Discovering your archive

Before Foundry can understand your archive, it has to discover it.

Imagine pointing it at five years of work spread across active projects, old client folders, exports, and external drives.

```text
Projects/
Clients/
Archive/
External_HDD/
Downloads/
```

Thousands of files appear: PSDs, Blender scenes, Premiere projects, audio stems, PDFs, references, screenshots.

Most teams think they know the answer until they scan a production drive for the first time.

The first version of Foundry just walks the folder tree and records facts. No ranking, no model calls, no search UI. It creates a manifest.

In game production, this surfaces what is spread across level directories, outsource drops, and archive branches. In film post, it maps proxies, source footage, and exports that live in different storage paths. In design and animation, it reveals how much material exists outside the current project folder but still affects active work.

The first manifest can be simple:

```text
filename
path
size
modified_date
file_type
hash
```

Nothing clever yet. Just a trustworthy inventory.

## Phase 2 — Giving files meaning

Once we know what exists, we enrich it.

Imagine opening a folder and seeing this:

```text
final.psd
final_FINAL.psd
final_v5.psd
export_new_FINAL.png
```

Those names carry almost no usable context. The files themselves carry a lot: mood, palette, composition, typography, subject matter, style.

An image can receive a generated description. A document can be summarized. A video can produce keyframe-level descriptors and transcript text. Audio can be transcribed and labeled with structural notes.

So this:

```text
final_v7.psd
```

becomes a richer record with usable context.

In production terms, this is where retrieval stops depending only on filenames and starts depending on observable content.

For a design team, that might include campaign and status fields. For animation, sequence and shot metadata. For game teams, project branch and engine compatibility. For film, rights and delivery constraints.

Metadata is still critical here. Meaning helps with recall. Metadata enforces precision.

## Phase 3 — Creating embeddings

Even with enrichment, pure text matching still misses intent shifts.

A user searching for spooky woodland may need the same material another person described as dark fantasy forest. Text overlap can be weak while meaning overlap is high.

Think of an embedding as a coordinate on a very large map of meaning. Two dark-forest concepts can appear close together on that map even if one file is named forest.psd and the other is named final_v7.png. Instead of searching only words, we start searching concepts.

This is where embeddings enter the flow. We convert descriptions and extracted text into vectors, then store them in Weaviate with metadata.

At this point, we have moved from files plus labels to a searchable semantic index that can handle language variation across teams.

---

## Why this is not another file search

Spotlight, Finder, and Windows Search are useful and should remain part of your workflow.

They are designed primarily for exact filenames, paths, and literal text matches. Foundry solves a different retrieval task.

Traditional file search asks: which file contains these exact words?

Foundry asks: which asset is most similar to what I am describing?

That is why semantic retrieval belongs alongside existing search, not in opposition to it.

## Phase 4 — Searching with hybrid retrieval

Now we add the actual query workflow: a search bar, production filters, and relevance-ranked results.

Hybrid search combines keyword signals and vector similarity in one request. That combination is important in real projects because users move constantly between exact and fuzzy intent.

Example patterns:

- A music producer remembers the exact sample pack name: keyword signal dominates.
- A game artist remembers mood and composition but not file names: semantic signal dominates.
- A film editor needs a mood plus a rights constraint: semantic plus metadata filters.

What this can feel like in practice:

```text
Search:
dark fantasy forest with blue fog

---------------------------------

Results

1) DarkForestConcept_v3.psd
	97% match
	Project: Nightfall

2) CastleMoodBoard.png
	94% match
	Project: RPG_Prototype

3) FogPalette.ai
	90% match
	Project: LightingTests
```

This is where Foundry becomes operational, not conceptual.

## Phase 5 — Teaching Foundry

Good retrieval systems are not static. Teams should be able to mark results as useful or not useful and feed that back into ranking and data quality improvements.

This can start small:

- thumbs up and thumbs down on results
- saved query sets by role or project
- regular review of failed queries

Creative vocabulary shifts over time. Feedback is how the system keeps up.

---

## What Foundry will not do

Foundry is intentionally narrow. It is not a replacement for storage, source control, or asset management systems.

```text
Foundry will not replace:
- Dropbox
- Google Drive
- NAS storage
- Perforce
- Git
- DAM platforms
```

It adds a retrieval layer on top of what teams already use.

---

## Technology stack

Foundry stays intentionally simple.

- Desktop app shell: Electron or Tauri
- Scanner: Node.js filesystem pipeline
- Enrichment: vision model, OCR, and audio transcription
- Storage and retrieval: Weaviate
- Frontend: React
- Search mode: hybrid search with metadata filters

This is not the only possible architecture, but it is practical for teams that want to build quickly and iterate in public.
